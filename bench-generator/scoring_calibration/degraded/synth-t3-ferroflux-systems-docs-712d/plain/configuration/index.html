Configuration DSL

FerroFlux nodes are configured through .flux files — a strict, strongly-typed configuration DSL that compiles to an immutable runtime manifest at boot. This page documents the four top-level blocks, shows annotated examples, and provides a searchable reference of all 47 configuration keys.

File location
The daemon loads /etc/ferroflux/node.flux by default, then merges any files matching /etc/ferroflux/conf.d/*.flux in lexical order. Use ferroflux config check to validate before deploying.

1. Global Settings

The global block defines node identity, logging behavior, and runtime limits. Exactly one global block is required per configuration tree; subsequent definitions raise E_DUP_GLOBAL at parse time.

Identity fields (node_id, region) become immutable once the node has registered with a broker — changing them requires a full re-pairing. Most other keys hot-reload on SIGHUP.

flux 
# /etc/ferroflux/node.flux — global section
global {
  node_id      = "edge-pdx-014"      # unique within tenant
  region       = "us-west-2"
  log_level    = info                 # trace|debug|info|warn|error
  log_format   = json
  metrics_port = 9100
  workers      = 8                    # tokio runtime threads
  queue_depth  = 4096
  tls_min      = "1.3"
}

2. Broker Profiles

A broker block declares a connection to an upstream FerroFlux Gateway (or compatible MQTT 5 / AMQP 1.0 endpoint). You may declare multiple profiles; the runtime fails over in declaration order when the primary becomes unreachable for more than failover_grace seconds.

Credentials are referenced indirectly via the secret_ref field; the literal secret field DEPRECATED since v2.2 and emits a parse warning. It will be removed in v3.0.

flux 
broker "primary" {
  endpoint        = "tls://gateway.ferroflux.io:8883"
  protocol        = mqtt5
  tenant          = "acme-industrial"
  secret_ref      = "vault:kv/ferroflux/primary"
  keepalive_secs  = 30
  qos             = 1
  failover_grace  = 12
}

broker "fallback" {
  endpoint        = "tls://gateway-eu.ferroflux.io:8883"
  protocol        = mqtt5
  tenant          = "acme-industrial"
  secret_ref      = "vault:kv/ferroflux/fallback"
}

3. Device Schemas

Device schemas describe the shape of telemetry payloads that a sensor class is expected to publish. The compiler turns each schema block into a zero-copy validator backed by serde + prost; payloads that fail validation are dropped and surface as schema_violations_total in the metrics endpoint.

Field types: f32, f64, i32, i64, u32, u64, bool, string(N), enum(...), and the composite vec3 for accelerometer triples.

flux 
schema "thermal_sensor_v3" {
  version     = 3
  encoding    = protobuf            # protobuf|cbor|json

  field temp_c      : f32    { min=-40.0, max=150.0, required=true }
  field humidity    : f32    { min=0.0,   max=100.0 }
  field battery_pct : u32    { max=100 }
  field mode        : enum("idle", "active", "calibrate")
  field ts          : i64    { required=true, monotonic=true }
}

4. Retention Policies

Retention policies bind to streams (matched by glob) and control how long raw and downsampled data remain on disk before being rolled up or evicted. A node may define any number of policies; the first matching glob wins.

Caution
Shrinking a retention window does not retroactively delete data; use ferroflux retention compact --since <ts> to force a sweep.
flux 
retention "hot-thermal" {
  match           = "streams/thermal/*"
  raw_window      = "7d"
  rollup_window   = "90d"
  rollup_interval = "5m"
  aggregations    = [min, max, avg, p95]
  storage_tier    = ssd
}

retention "cold-vibration" {
  match           = "streams/vibration/*"
  raw_window      = "24h"
  rollup_window   = "365d"
  rollup_interval = "1h"
  storage_tier    = archive
}
