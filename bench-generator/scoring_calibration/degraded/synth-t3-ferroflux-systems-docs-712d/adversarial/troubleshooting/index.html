Troubleshooting

Diagnostic workflows for the four most commonly reported failure modes when operating FerroFlux gateways and embedded sensor fleets. Each section opens with a symptom table, followed by ordered resolution steps and the exact commands required to verify a fix.

Before you begin. Confirm your gateway is running SDK v2.3.0 or later. Many of the diagnostic commands below depend on the fluxctl diag subcommand introduced in 2.3.

1. Connection Failures

Connection failures manifest when a registered device cannot establish or maintain a session with the gateway broker. The middleware will surface these in the device.session stream with non-zero exit codes.

SymptomLikely CauseSeverity
ECONNREFUSED on port 8443 Broker process not listening or firewall dropping inbound TLS. CRITICAL
handshake_timeout (30s) Mismatched TLS cipher suites or expired client certificate. HIGH
flapping reconnects > 5/min Keepalive interval shorter than network MTU round-trip. MEDIUM
SESSION_REJECTED_FULL Broker session pool exhausted; raise max_sessions. LOW

Resolution Steps

  1. Verify the broker is reachable.

    Probe the broker's TLS listener from the device subnet. A healthy broker returns the server certificate chain within 2s.

    bash 
    # From the device-side network namespace
openssl s_client -connect broker.flux.local:8443 \
  -servername broker.flux.local -showcerts < /dev/null
  2. Inspect the session log.

    Use fluxctl diag sessions to enumerate the last 50 session transitions for a device. Look for a repeated HANDSHAKE_FAIL reason.

    bash 
    fluxctl diag sessions --device "edge-007" --tail 50
  3. Rotate the device certificate.

    If the certificate is within 14 days of expiry, rotate using the SDK's renewal helper. The device will reconnect automatically once the new cert lands in its keystore.

    rust 
    use ferroflux::auth::CertRenewal;

let renewal = CertRenewal::new(&ctx)
    .with_csr_path("/etc/flux/csr.pem")
    .execute()
    .await?;

println!("renewed: {}", renewal.fingerprint);
  4. Tune keepalive.

    Set keepalive_secs in the device profile to at least 2× the observed network RTT. Default is 30; raise to 60–90 for cellular fleets.

2. Schema Validation Errors

Every reading pushed into FerroFlux is validated against the device's declared schema. Mismatches are rejected at ingest and never reach downstream streams.

SymptomLikely CauseSeverity
SCHEMA_FIELD_MISSING Required field absent from payload; check schema version pinning. HIGH
SCHEMA_TYPE_MISMATCH Floating-point sent where integer expected (or inverse). MEDIUM
SCHEMA_LEGACY_V1 DEPRECATED Device still emitting v1 envelopes. Migrate before SDK 3.0. MEDIUM
SCHEMA_RANGE_VIOLATION Numeric reading outside min/max declared range. LOW

Resolution Steps

  1. Print the active schema.

    Confirm which schema revision the gateway is enforcing for this device.

    bash 
    fluxctl schema show --device "edge-007" --format json
  2. Dry-run a payload locally.

    Validate a sample reading against the schema without sending it.

    bash 
    fluxctl schema validate \
  --schema sensors/temp_v3.flux \
  --payload '{"ts":1719283200,"c":21.4}'
  3. Migrate from v1 envelopes.

    Wrap legacy readings using the compatibility shim until firmware can be reflashed.

    rust 
    use ferroflux::compat::v1_envelope;

let reading = v1_envelope::lift(legacy_bytes)?;
client.push("streams/temp", &reading).await?;

3. High Latency Readings

End-to-end latency is measured from device.ts to ingest.ts. The platform SLO is <800ms at the 95th percentile.

SymptomLikely CauseSeverity
p95 latency > 2s sustained Batch buffer too large; reduce flush_threshold. HIGH
clock_skew > 5s Device NTP drift; falls back to broker timestamp. MEDIUM
backpressure_dropped Ingest queue saturated; horizontally scale broker shards. CRITICAL

Resolution Steps

  1. Inspect ingest latency histogram.
    bash 
    fluxctl diag latency --window 15m --bucket p50,p95,p99
  2. Lower the device flush threshold.

    In the device's .flux profile, drop flush_threshold from 256 to 32 readings to trade throughput for freshness.

    flux 
    profile "edge-fast" {
  flush_threshold = 32
  flush_interval  = "500ms"
  compression     = "zstd"
}
  3. Resync NTP.

    Force a chrony resync and verify drift is under 200ms.

    bash 
    chronyc -a makestep
chronyc tracking | grep "System time"

4. Authentication Rejections

FerroFlux authenticates devices via mutual TLS and operators via short-lived JWTs issued by /auth/token.

SymptomLikely CauseSeverity
401 invalid_signature Operator JWT signed with rotated key. HIGH
403 device_revoked Device fingerprint present in revocation list. CRITICAL
AUTH_BASIC_HEADER DEPRECATED Basic auth removed in v2.2. Migrate to bearer tokens. HIGH
429 too_many_logins Login rate-limit; back off for 60s. LOW
Deprecation notice. Basic authentication headers will return 401 in SDK v3.0. Issue bearer tokens via POST /auth/token and pass them in Authorization: Bearer ….

Resolution Steps

  1. Rotate the operator signing key.
    bash 
    fluxctl auth rotate-key --kid "ops-2024-q3"
  2. Verify the device fingerprint.
    bash 
    fluxctl devices show "edge-007" | grep fingerprint
# Compare to expected SHA-256 in your CMDB
  3. Re-enroll if revoked in error.

    Remove the entry from the revocation list and re-issue a certificate. The device must reboot to clear cached rejection state.

Collecting Diagnostic Logs

When opening a support ticket, include a full fluxctl diag bundle archive. The bundle contains 24h of broker logs, device session traces, schema cache state, and the active configuration — all redacted of secrets.

bash 
# Generate a redacted diagnostic bundle
fluxctl diag bundle \
  --since "24h" \
  --include sessions,schema,config,metrics \
  --redact secrets,pii \
  --output "/tmp/flux-diag-$(date +%s).tgz"

Attach the resulting tarball to your ticket along with the affected device IDs and the approximate UTC timestamp of the first observed failure. Support response targets are 4 business hours for CRITICAL and 1 business day for everything else.

Tip. You can stream live diagnostic events to your terminal with fluxctl diag tail --level=warn. Useful while reproducing intermittent issues.
