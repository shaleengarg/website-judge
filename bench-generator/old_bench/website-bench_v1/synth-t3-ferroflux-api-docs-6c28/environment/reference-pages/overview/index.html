Overview

Real-time sensor data ingestion, streaming, and analytics — built for industrial-scale telemetry.

FerroFlux is a managed platform for ingesting, buffering, and querying high-frequency sensor data from industrial fleets, IoT devices, and edge gateways. The API exposes durable data streams, sub-second query latency, and a uniform SDK surface across Python, Node.js, Go, and Rust. Whether you're piping a single vibration sensor or tens of thousands of pressure transducers, the same primitives apply.

This documentation is organized into a Quickstart guide, a complete REST endpoint reference, the SDK reference, and the changelog. New users should start with the Quickstart.

Core Features

⚡

Sub-Second Ingestion

Push up to 50k samples/sec per stream with end-to-end latency under 200ms across regions.

🔐

Signed Streams

Per-device HMAC keys, scoped tokens, and rotating secrets keep your telemetry tamper-evident.

📊

Live Aggregations

Built-in windowed averages, percentiles, and anomaly flags computed at write time, not query time.

🌐

Multi-Region

Active-active replication across us-east, eu-west, and ap-south with automatic failover.

🔌

Edge SDKs

Lightweight clients for Python, Node.js, Go, and Rust — plus an MQTT bridge for legacy devices.

📦

Cold Storage Tier

Automatic archival of streams older than 30 days to Parquet on S3-compatible object stores.

Architecture

The FerroFlux pipeline is built around a write-optimized log, a real-time query layer, and a downstream analytics tier. Devices authenticate at the edge, push to the nearest regional ingest gateway, and data becomes queryable within a single round-trip.

A Minimal Example

The snippet below creates a stream and publishes a single reading using the Python SDK:

Python# pip install ferroflux
from ferroflux import Client

client = Client(api_key="ff_live_••••")
stream = client.streams.create(name="turbine-07.vibration")

stream.publish({
    "ts": "2025-03-14T09:21:00Z",
    "value": 12.7,
    "unit": "mm/s"
})

Key Concepts

Streams

A stream is an append-only, ordered channel of timestamped samples. Each stream has a stable ID, a schema descriptor, and a retention policy. Streams are the primary unit of authorization.

Readings

A reading is a single sample: a timestamp, a numeric or structured value, and optional tags. Readings are immutable once acknowledged by the ingest gateway.

Projections

A projection is a continuously-maintained derived view — for example, a 1-minute rolling P95 over a vibration stream. Projections are defined declaratively and queried like any other stream.

  • Streams are immutable; readings cannot be edited, only superseded.
  • Projections are computed at write time and cached in the hot tier for 30 days.
  • All timestamps are RFC 3339 UTC with nanosecond precision.
